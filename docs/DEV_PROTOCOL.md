## Протокол сверки с источниками (перед каждым Phase 2)

Перед финализацией плана и написанием кода — обязательная сверка со всеми источниками истины проекта:

```
а) Диаграммы   — TO_BE, AS_IS, ER (drawio/PNG)
                 → проверить что фича не противоречит бизнес-процессу
                 → файлы: docs/diagrams/

б) Дизайн-доки — 2.1_Color_Typography.md
                 → CSS-токены, Heroicons outline 24px, шрифт Inter
                 → все цвета только через CSS-переменные, нет raw HEX в компонентах

в) Кодовая база — реальный код, не предположения
                 → base.html      — структура шаблона, вложенность .container
                 → views.py       — существующие константы, импорты, view-функции
                 → models.py      — поля, FK, choices, class-level constants
                 → urls.py        — зарегистрированные маршруты и их имена

г) ТЗ / 1.4    — F01–F14, матрица приоритетов
                 → привязать каждую фичу к конкретному требованию
                 → файл: docs/1.4_Analiz_Trebovaniy_Polzovateley.md

д) README / ИЗ — тема ВКР, план текущей недели, Changelog
                 → фича должна быть в плане утверждённой недели
                 → файл: README.md

е) TZ_CHECKLIST — балльная шкала куратора (19 критериев, 57 баллов)
                 → для каждой задачи ответить на 3 вопроса:
                   1. Какой критерий из таблицы эта задача закрывает или затрагивает?
                   2. Не регрессирует ли уже закрытый критерий (✅)?
                   3. Есть ли ⚠️-блокеры, которые нужно закрыть до деплоя?
                 → после Phase 3: обновить статус строки в TZ_CHECKLIST.md и пересчитать счёт
                 → файл: docs/TZ_CHECKLIST.md

ж) ИЗ преподавателя — жёсткие требования из индивидуального задания
                 → источник истины #1, нарушать нельзя
                 → ключевые строки для веб-сайта:
                   - ≥10 страниц для неавторизованного пользователя
                   - макет каждой страницы
                   - ≥10 пунктов меню + хлебные крошки на каждой странице
                   - БД ≥10 таблиц, PostgreSQL, дамп в репозитории
                   - ≥3 роли пользователей
                   - Панель администратора ≥5 страниц
                   - Личный кабинет ≥5 страниц (для каждой роли)
                   - Доступ к файловой системе
                   - Формирование .docx / .xlsx
                   - ≥2000 логических строк кода
                   - ≥50 коммитов, распределённых по времени
                 → проверять: не противоречит ли фича этим требованиям?
                 → оригинал: строки 115–124 файла ИЗ (uploads/17. Люлюкин_ИЗ.docx)
```

Если по любому пункту найдено противоречие — фиксируем в **Problems Found** (Phase 2, п. 2.3) и решаем до кода.

## Правила hardcode (абсолютные, без исключений)

- Числовые лимиты → именованные константы на уровне модуля (`HOME_SERVICES_COUNT`, `SERVICES_PER_PAGE`)
- Строковые роли/статусы → class-level constants (`ROLE_CLIENT`, `Project.STATUS_NEW`)
- Списки статусов → именованные константы (`Project.PORTFOLIO_STATUSES`)
- Цвета в CSS → только `var(--token)`, нет `#HEX` в компонентных блоках
- URL-имена → только `{% url 'core:name' %}`, нет строк-путей вручную

---

## Правила шаблонов

- `base.html` оборачивает `{% block content %}` в `.container` — **не добавлять** inner `.container` в шаблонах страниц (Tech Debt до отдельного коммита)
- Каждая `<section>` → `aria-labelledby="<id>"`
- Списки → `role="list"` на `<ul>`
- Empty state → `role="status"`
- Иконки → `aria-hidden="true"`
- Даты → `<time datetime="YYYY-MM-DD">`

## Порядок коммита

1. Phase 1 — анализ (протокол сверки а–е, включая TZ_CHECKLIST)
2. Phase 2 — план → **СТОП, ждём подтверждения**
3. Phase 3 — код
4. Проверки: AST, URL-теги, hardcode, HEX
5. DevLog → `docs/DevLog_ВКР-NNN_YYYY-MM-DD.md`
6. RELEASE_NOTES.md → prepend новый релиз
7. TZ_CHECKLIST.md → обновить статус закрытых критериев + пересчитать счёт
8. README.md → только при закрытии недели (не после каждого коммита)

## Шаблон строки Phase 1 → TZ_CHECKLIST (вставлять в каждый DevLog)

```
| е) TZ_CHECKLIST | Критерий #N «...» — [закрываем ✅ / в работе 🔄 / не затрагиваем]

|                 | Активные ⚠️-блокеры: перечислить или «нет»
```
